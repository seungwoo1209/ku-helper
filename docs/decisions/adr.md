# Architecture Decision Record (ADR)

## 1. 기술 스택 선정 및 근거

### 프론트엔드 — React 18+

본 시스템의 웹 대시보드는 사용자가 알림 조건을 동적으로 추가·수정·삭제하는 인터랙티브한 UI를 요구한다. 알림 항목별로 상태가 독립적으로 관리되어야 하고, 설정 변경 시 페이지 전체를 새로고침하지 않고 해당 컴포넌트만 즉시 반영되어야 하므로, 컴포넌트 단위의 상태 관리와 가상 DOM 기반의 효율적 렌더링을 제공하는 React가 적합하다. 또한 React 18에서 도입된 Concurrent Rendering은 알림 목록이 많아질 경우에도 UI 블로킹 없이 렌더링을 처리할 수 있어 사용자 경험 측면에서 유리하다.

### 백엔드 API — FastAPI (Python 3.11+)

백엔드는 지하철 공공 API 호출, 학식 크롤링, 도서관 좌석 조회 등 I/O 바운드 작업이 빈번하게 발생하는 구조다. FastAPI는 Python의 `async/await` 기반 비동기 처리를 네이티브로 지원하여, 여러 외부 API를 동시에 호출하더라도 스레드를 블로킹하지 않고 효율적으로 처리할 수 있다. 또한 Pydantic 기반의 자동 요청/응답 유효성 검사와 OpenAPI 문서 자동 생성 기능은 팀 내 API 명세 공유 및 프론트엔드와의 연동 과정에서 커뮤니케이션 비용을 줄여준다. APScheduler, asyncIO Queue 등 Python 생태계 라이브러리와의 통합도 자연스러워 단일 언어로 백엔드 전반을 관리할 수 있다는 점도 소규모 팀에 적합한 판단이었다.

### 디스코드 봇 — discord.py 2.0+

본 시스템의 핵심 출력 채널이 디스코드 DM인 만큼, 디스코드 API와의 통합 완성도가 중요하다. discord.py 2.0은 `asyncio` 기반으로 설계되어 FastAPI 백엔드와 동일한 비동기 실행 모델을 공유하며, 임베드 메시지 구성, DM 채널 생성, OAuth2 사용자 식별 등 본 프로젝트에서 필요한 기능을 공식 API 수준으로 지원한다. 특히 Discord OAuth2와의 연동을 통해 사용자의 Discord ID를 직접 식별할 수 있어, 별도 회원가입 없이 봇이 정확한 대상에게 DM을 발송하는 흐름을 구현할 수 있다.

### 인증 — Discord OAuth2 + JWT

사용자가 별도 계정을 만들지 않고 기존 디스코드 계정으로 로그인할 수 있도록 하는 것이 본 시스템의 사용 편의성 목표와 일치한다. Discord OAuth2 Authorization Code Grant 방식을 통해 사용자의 Discord ID를 안전하게 획득하고, 봇은 해당 ID를 이용하여 사용자와 봇이 같은 서버에 속하면 DM을 발송한다. 이후 JWT를 발급하여 세션 상태를 서버에 저장하지 않는 Stateless 방식으로 인증을 유지함으로써, 추후 서버를 수평 확장하더라도 세션 동기화 문제가 발생하지 않는 구조를 갖춘다.

### 스케줄러 — APScheduler

지하철 도착 정보나 학식 메뉴는 정해진 주기로 디스코드 DM으로 알림을 보내 정보가 갱신되어야 하고, 사용자가 설정한 알림 시각에 맞춰 정기적으로 조건 판단 로직을 실행해야 한다. APScheduler는 Python 프로세스 내에서 크론 표현식 기반의 정기 작업을 별도 인프라 없이 간단하게 등록할 수 있다. discord.py 봇 프로세스와 동일한 asyncio 이벤트 루프에서 동작하여, 봇 기동 시 자동으로 스케줄이 등록되고 별도 프로세스 관리가 필요 없다는 점도 선택 이유다.

### 캐시 — Redis

지하철 도착 정보나 학식 메뉴는 동일한 시간대에 여러 사용자가 조회하더라도 데이터가 동일하다. 매 요청마다 외부 API를 호출하면 응답 지연 및 API 호출 횟수 제한 문제가 발생할 수 있으므로, Redis에 일정 TTL로 응답을 캐싱하여 불필요한 중복 호출을 방지한다. 동시에 Redis는 state token 저장 용도로도 활용되어, 단일 Redis 인스턴스가 여러 역할을 수행하며 인프라 복잡도를 최소화한다.

## 2. 설계 결정 근거

| 알림 전달 채널 | Discord DM 봇을 통한 알림 발송 | 이메일, SMS, 카카오톡, 자체 푸시 | 대학생 주요 사용 플랫폼이 Discord임을 전제로, 별도 앱 설치 없이 기존 사용 환경에서 알림 수신 가능. OAuth2 인증과 DM 발송을 단일 플랫폼에서 처리할 수 있어 외부 연동 비용 최소화 | FR-001, FR-019 |
| 사용자 인증 방식 | Discord OAuth2 + JWT | 자체 회원가입, Google OAuth, 카카오 OAuth | 알림 수신 채널이 Discord인 이상, 사용자의 Discord ID를 가장 신뢰성 있게 획득할 수 있는 수단이 Discord OAuth2임. 자체 회원가입 대비 사용자 진입 장벽을 낮추고, JWT로 Stateless 인증을 유지하여 서버 수평 확장 시 세션 동기화 문제를 방지 | FR-001, NFR-005, NFR-014 |
| 데이터베이스 | PostgreSQL + JSONB | MySQL, MongoDB, SQLite | 사용자-알림 설정 간 관계형 무결성은 RDB로 관리 가능, 알림 유형별로 구조가 다른 세부 조건은 JSONB로 유연하게 저장. 알림 유형 확장 시 스키마 변경 없이 조건 필드를 추가할 수 있으며, JSONB 인덱스로 조회 성능도 확보 가능 | FR-006, FR-013, NFR-014 |
| 백엔드 프레임워크 | FastAPI (Python) | Django, Spring Boot, Express.js | 지하철 공공 API 호출, 학식 크롤링, 도서관 좌석 조회 등 I/O 바운드 작업이 빈번한 구조에서, Python async/await 기반 비동기 처리를 네이티브로 지원하는 FastAPI가 적합. Pydantic 자동 유효성 검사와 OpenAPI 문서 자동 생성으로 프론트엔드와의 협업 비용 절감 효과 또한 기대함 | NFR-001, NFR-002, NFR-008 |
| 알림 발송 비동기 처리 | asyncio Queue | FastAPI Background Tasks, Celery | 다수 사용자에게 동시 DM 발송 시 Discord API Rate Limit(초당 1건) 초과를 방지하기 위해 발송 작업을 큐에 적재하고 워커가 순차 처리하는 구조 필요. Celery는 학습 곡선으로 인해 프로젝트에 적합하지 않다고 판단함 | FR-021, NFR-003, NFR-002 |
| 캐시 | Redis | Memcached, RabbitMQ (브로커 분리) | API 응답 캐싱, State Token 관리를 단일 Redis 인스턴스로 통합하여 인프라 복잡도를 최소화. TTL 기반 캐시 만료로 외부 API 장애 시 최근 캐시 데이터를 활용한 알림 발송이 가능 | NFR-010, NFR-002 |
| 스케줄링 방식 | APScheduler | Celery Beat, Cron, AWS EventBridge | 크롤링 주기 실행과 정기 알림 스케줄링을 별도 인프라 없이 FastAPI 프로세스 내에서 크론 표현식 기반으로 등록 가능. 초기 구현 복잡도를 낮추면서 서버 기동 시 자동으로 스케줄이 등록되는 구조로 운영 편의성 확보 | FR-007, FR-011, NFR-004 |
| 알림 조건 저장 구조 | JSONB 단일 필드로 조건 저장 | 알림 유형별 별도 테이블 분리 | 교통·점심·도서관 알림의 세부 조건 구조가 각기 다르고, 향후 알림 유형 추가 가능성이 있음. 유형별 테이블 분리 시 알림 추가마다 스키마 변경이 필요하나, JSONB 구조는 조건 필드를 자유롭게 확장할 수 있어 유지보수성이 높음 | FR-006, FR-013, NFR-011 |
| 중복 알림 방지 전략 | 발송 이력 + 상태 플래그 기반 | 단순 시간 기반 쿨다운 | 도서관 좌석 알림의 경우 단순 시간 쿨다운 방식은 좌석이 회복되지 않았음에도 재발송되는 문제가 발생. 잔여석이 임계값 이상으로 회복된 후 재하락했을 때만 알림을 발송하는 상태 기반 방식으로 실질적 중복을 차단 | FR-014, NFR-002 |
| 컨테이너화 전략 | Docker + Docker Compose | 직접 설치(Bare Metal), Kubernetes | FastAPI, Discord 봇, Redis, PostgreSQL 등 다수 서비스를 단일 팀이 관리하는 구조에서, Docker Compose로 전체 스택을 하나의 설정 파일로 정의하여 개발·운영 환경 일관성을 확보. 현재 단일 대학 규모에서 Kubernetes 수준의 오케스트레이션은 과도한 복잡도를 초래 | NFR-009, NFR-014 |
| 크롤링 아키텍처 | 모듈별 독립 크롤러 | 단일 통합 크롤러 | 학식·도서관 크롤링 대상 웹사이트의 구조 변경 시 해당 모듈만 수정하고 재배포할 수 있도록 크롤러를 독립 모듈로 분리. 단일 크롤러 구조는 하나의 대상 변경이 전체 크롤링 로직에 영향을 미치는 위험이 있음 | NFR-011, FR-022 |
| 프론트엔드 프레임워크 | React 18+ | Vue.js, Next.js, 순수 HTML/CSS | 알림 조건별 독립적인 상태 관리와 설정 변경 시 해당 컴포넌트만 즉시 반영하는 동적 UI가 필요. React 18의 Concurrent Rendering은 알림 목록이 많아지더라도 UI 블로킹 없이 처리 가능하며, 반응형 대시보드 구현에 적합한 컴포넌트 생태계를 보유 | FR-016, INT-002 |
