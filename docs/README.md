## 1. 시스템 개요 기술

### 1.1 시스템이 해결하는 핵심 문제

대학생들은 학식 메뉴 확인, 지하철 도착 시간 조회, 도서관 좌석 현황 확인 등 매일 반복적으로 유사한 정보를 탐색한다. 그러나 이러한 정보들은 서로 다른 웹사이트나 애플리케이션에 분산되어 있어, 사용자는 각각의 서비스를 개별 접속해야 하는 불편함이 존재한다.

본 프로젝트는 이러한 반복 탐색하는 정보(지하철 도착 시간, 학식 메뉴, 도서관 좌석 현황)를 하나의 웹 대시보드에서 알림 조건으로 설정하면, 디스코드 봇이 개인 메시지(DM) 형태로 자동 전달하는 시스템을 구축하는 것을 목적으로 한다. 이를 통해 사용자는 별도 앱 설치 없이 기존에 사용 중인 디스코드 플랫폼 내에서 맞춤형 정보를 효율적으로 수신할 수 있다.

### 1.2 시스템의 주요 사용자 유형

본 시스템의 주요 사용자는 매일 반복적으로 캠퍼스 관련 정보를 탐색하는 대학생을 중심으로 하며, 다음과 같은 유형으로 세분화할 수 있다.

* 정보 효율 추구형 사용자 : 학식 메뉴, 도서관 좌석, 지하철 도착 시간 등 여러 정보를 매일 개별 앱이나 웹사이트에서 직접 확인하는 데 피로감을 느끼는 학생이다. 이들은 정보 접근 과정을 단순화하고자 하는 동기가 강하며, 본 시스템의 핵심 수혜 대상이다.
* 일정 민감형 사용자 : 강의 시작 시간, 지하철 배차 간격 등 시간에 민감한 정보를 놓쳤을 때 실질적인 불이익이 발생하는 학생이다. 이들은 알림 조건 설정 기능을 적극적으로 활용하여 특정 시각에 맞춤형 정보를 수신하는 방식을 선호한다.
* 디스코드 상시 사용자 : 학업 및 동아리 활동 등으로 디스코드를 이미 일상적으로 사용하고 있는 학생이다. 별도 앱 설치나 새로운 플랫폼 학습 없이 기존 환경 안에서 정보를 수신할 수 있다는 점에서 진입 장벽이 낮고 지속 사용 가능성이 높다.
* 정보 수집 수동형 사용자 : 필요한 정보가 있어도 직접 검색하는 행동을 번거롭게 여겨 정보 확인 자체를 생략하거나 미루는 경향이 있는 학생이다. 이들에게는 시스템이 능동적으로 정보를 DM으로 전달한다는 점이 핵심적인 사용 유인으로 작용한다.

### 1.3 시스템 범위 (Scope)

**In Scope (포함 범위)**

* 디스코드 OAuth2 소셜 로그인
* 교통 알림: 서울 공공 API 기반 지하철 실시간 도착 정보 알림
* 점심 알림: 학식 메뉴 크롤링 및 주변 음식점 추천
* 도서관 알림: 잔여 좌석 임계값 기반 알림
* 웹 설정 대시보드 (반응형)
* 디스코드 봇 DM 임베드 메시지 전송
* 평일/주말 구분 알림 및 활성 시간대 설정
* 알림 발송 이력 관리 및 중복 알림 방지
* 신뢰성 체계 구축(크롤러 로그 체계 구축, DM·크롤러 실패 시 관리자 알림)

**Out of Scope (제외 범위)**

* 버스 알림
* 도서관 정기 현황 알림
* 점심 카테고리 필터 / 유명 맛집 비율 조절
* 사용자 리뷰 평점 시스템
* 카카오톡, 텔레그램 등 추가 알림 채널
* 학사 공지, 날씨, 시험 일정 알림
* 다대학 확장 (다중 캠퍼스 지원)
* 모바일 네이티브 앱 / PWA

## 2. 아키텍처 설계 (Architecture Design)

### 2.1 시스템 구성요소 다이어그램

#### 2.1.1 시스템 아키텍처

본 시스템은 AWS 단일 리전 내에 구성되며, VPC를 통해 네트워크를 격리하고 EC2 인스턴스 위에 Docker Compose 기반으로 전체 서비스를 컨테이너화하여 운영한다.

##### 외부 진입 경로

사용자는 React SPA 기반 웹 대시보드를 통해 서비스에 접근한다. 최초 로그인 시 Discord OAuth2 Authorization Code Grant 방식으로 인증을 수행하며, Discord 인증 서버에서 발급된 Authorization Code는 EC2 내부의 API Server로 전달되어 사용자의 Discord ID 획득 및 JWT 토큰 발급에 사용된다. 이후 모든 API 요청은 Nginx 리버스 프록시를 경유하여 API Server에 도달한다. Nginx는 HTTPS 종단 처리, 정적 파일 서빙, 그리고 API 요청의 리버스 프록시 역할을 수행한다.

##### EC2 내부 구성

EC2 인스턴스 내에는 크게 두 개의 애플리케이션 컨테이너가 존재한다.

첫째, FastAPI 기반 **API Server**는 사용자 인증, 알림 설정 CRUD, 알림 이력 조회 등 RESTful API를 제공한다. 사용자가 웹 대시보드에서 설정한 알림 조건은 이 서버를 통해 PostgreSQL에 JSONB 형태로 저장된다.

둘째, **Bot 컨테이너**는 APScheduler, asyncio.Queue, discord.py가 하나의 Python 프로세스 내에서 동작하는 구조이다. APScheduler가 주기적으로 PostgreSQL을 폴링하여 활성화된 알림 설정을 감지하고, 데이터 수집(공공 API 호출 및 웹 크롤링)과 조건 판단을 수행한다. 조건이 충족되면 asyncio.Queue에 발송 작업을 적재하고, 발송 워커 코루틴이 Discord API Rate Limit에 맞춰 순차적으로 discord.py를 통해 사용자에게 DM 임베드 메시지를 전송한다.

두 컨테이너는 직접 통신하지 않으며, PostgreSQL을 매개로 간접적으로 연결된다.

##### 데이터 계층

RDS(PostgreSQL)는 사용자 정보, 알림 설정, 발송 이력 등 영속 데이터를 저장한다. ElastiCache(Redis)는 공공 API 응답 캐싱, 크롤링 결과 캐싱, 중복 알림 방지 쿨다운, OAuth2 CSRF state 토큰 임시 저장 등 TTL 기반 임시 데이터 관리에 사용된다. 두 서비스 모두 VPC 내부에 위치하여 외부 접근이 차단된다.

##### 모니터링

**CloudWatch**는 리전 수준에서 EC2 내 Bot 컨테이너의 크롤러 실행 로그를 수집하고 모니터링하는 데 사용된다. 크롤링 대상 웹사이트의 구조 변경이나 공공 API 통신 장애 등으로 인한 수집 실패 패턴을 로그 기반으로 추적할 수 있다. 이와 별도로, 크롤러 연속 실패 시 관리자에게 Discord DM으로 즉시 알림을 발송하는 내부 장애 감지 체계를 병행 운영하여 신속한 장애 대응이 가능하도록 구성한다.
